Introduction
Overview
The TM HR Portal Design System defines the visual language, interaction patterns, and component library used across all modules of the GHCM HR Portal — including AskHR Ticketing, Employee Central, Skills Dashboard, and IDP.
🎨
Consistent Tokens
CSS variables define all colours, spacing, shadows and radii. Change once, apply everywhere.
🧩
Reusable Components
Every button, card, modal, and table follows the same structure and class conventions across all pages.
📐
Clear Patterns
Interaction patterns like filter bars, ticket flows, and audit trails are documented with live examples.
Introduction
Design Principles
🎯
Task-first, not feature-first
Every screen exists to help a user complete a task — raising a ticket, resolving an issue, or monitoring performance. Decorative complexity is avoided unless it aids comprehension.
🔍
Clear visual hierarchy
Navy anchors authority. Blue drives action. Orange signals primary CTAs. Status colours (green, yellow, red) are reserved strictly for states — never decorative use.
⚡
Immediate feedback
Every action should produce visible feedback — toast on submit, animated value update on filter change, status tag update on ticket action. Users should never wonder if something worked.
🔒
Role-aware UI
The interface surface adapts to the user's role (Employee, PIC, Super Admin). Components are shared; visibility and access are controlled by RBAC at runtime, not by building separate UIs.

Remarks Audit Trail
The audit trail is a read-only chronological log of all ticket events. No compose box — remarks are only added through Resolve or Revert modals (PIC) or the resubmit flow (Employee).

Submitted → Pending Review → In Progress → Pending Employee → In Progress → Resolved → Closed

Empty States
Empty states appear when a table has no results or a section has no data. Always include an icon, a clear message, and (where appropriate) a CTA to create something.
